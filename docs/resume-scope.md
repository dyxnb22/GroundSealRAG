# Resume Scope

## Purpose

This document explains how GroundSeal RAG can become a credible resume project without overstating its maturity.

## When It Becomes Resume-Worthy

The project becomes resume-worthy when it has more than documents. At minimum, it should demonstrate a working retrieval path, traceable citations, and evaluation results. The resume value comes from showing disciplined retrieval engineering, not from saying "built a RAG chatbot."

## Minimum Resume Scope

Suitable once the project includes:

- source and chunk metadata contracts
- lexical retrieval baseline
- citation-ready candidate output
- small evaluation set
- documented failure analysis

Possible description:

"Built a retrieval-first RAG learning project with source-aware chunking, lexical retrieval baseline, citation-ready evidence records, and a small evaluation workflow for analyzing retrieval failures."

## Recommended Resume Scope

Stronger once the project includes:

- lexical and semantic retrieval baselines
- hybrid retrieval comparison
- permission-aware filtering
- citation packing
- retrieval evaluation reports
- documented failure taxonomy

Possible description:

"Designed and implemented a permission-aware hybrid retrieval layer for grounded agents, comparing lexical and semantic baselines, enforcing access-scoped evidence filtering, producing citation-ready context packages, and evaluating retrieval quality with repeatable query cases."

## Enhanced Resume Scope

Strongest once the project includes:

- reranking experiments
- permission evaluation matrix
- citation quality evaluation
- CLI for repeatable experiments
- reports showing tradeoffs and failures
- optional generation layer that consumes evidence without bypassing retrieval

Possible description:

"Built an evaluation-driven retrieval system for grounded enterprise agents, including hybrid retrieval, permission-aware filtering, citation packing, reranking experiments, and repeatable evaluation reports covering recall, ranking, access correctness, and citation failures."

## How To Avoid Sounding Like A Generic RAG Demo

Emphasize:

- retrieval-first architecture
- permission scope filtering
- citation-first evidence packages
- evaluation cases and failure analysis
- baseline comparisons
- engineering contracts

Avoid leading with:

- chatbot interface
- prompt engineering
- model provider names
- vague enterprise-grade claims
- "answers questions over documents" as the main differentiator

## Capabilities Worth Highlighting

- retrieval system design
- metadata modeling
- permission-aware filtering
- hybrid retrieval tradeoffs
- citation traceability
- evaluation methodology
- failure analysis
- grounded context assembly
- disciplined phase-based execution

## Evidence To Preserve For Interviews

Keep artifacts that show reasoning:

- roadmap decisions
- evaluation reports
- failure records
- before and after retrieval comparisons
- examples of permission false allow and false deny prevention
- examples where citation packing changed the quality of context

The strongest interview story is not "the system worked." It is "the system exposed retrieval failures, measured them, and improved through controlled changes."
